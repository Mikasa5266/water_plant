# AGENTS.md

本文件约束 Codex、Claude 等 AI Agent 在本仓库内的行为。所有 Agent 在改动前必须先阅读本文件和 `docs/DEVELOPMENT_GUIDE.md`。

## 时间与知识时效

- 涉及框架、库、工具、API、配置、下载渠道、行业规则时，必须以真实当前日期为基准。
- 不得默认使用废弃 API、过时配置、失效链接、停维护工具或旧版教程。
- 如果版本差异会影响写法，必须优先采用当前稳定版官方写法，并说明新旧差异。
- 对时效性强的内容，优先核对官方文档或项目锁定版本。

## 工作边界

- 默认只修改用户明确要求范围内的文件。
- 不得因为“顺手优化”重构无关模块、移动目录、批量格式化无关文件。
- 不得编辑 `node_modules/`、`dist/`、缓存目录、构建产物、临时文件。
- 不得提交真实密钥、客户资料、硬件连接参数、生产地址或私有凭据。
- 未经明确要求，不得创建真实硬件控制逻辑；当前硬件操作只允许作为模拟、动画或接口占位。

## 目录边界

- 前端代码放在 `frontend/src/`。
- 后端代码放在 `backend/app/`。
- 前后端接口、Agent 事件、3D 控制事件等跨端契约放在 `contracts/`。
- 建模源文件放在 `assets/models/source/`。
- 运行时模型导出文件放在 `assets/models/exported/`。
- 项目规范、流程、方案文档放在 `docs/`。

## 关键文档

- `docs/DEVELOPMENT_GUIDE.md`: 总体协作规范。
- `docs/ARCHITECTURE.md`: 架构边界草案。
- `docs/FRONTEND_GUIDE.md`: 前端目录和代码规则。
- `docs/BACKEND_GUIDE.md`: 后端目录和安全规则草案。
- `docs/MODELING_GUIDE.md`: 3D 建模交付规则。
- `docs/GIT_WORKFLOW.md`: 分支、提交和 PR 规则。
- `contracts/README.md`: 接口、Agent 事件和 3D 事件契约入口。

## Agent 改动规则

- 改动前先查看当前文件结构和 Git 状态，避免覆盖他人工作。
- 大改动前先说明将修改哪些区域；发现已有未提交改动时必须保护它们。
- 新增依赖前必须确认必要性、维护状态和官方推荐用法。
- 修改接口字段、事件结构或模型资源路径时，必须同步更新 `contracts/` 或相关文档。
- 未经用户明确要求，不得执行 `git add`、`git commit`、`git push`、删除分支或重写历史。
- 如需删除、移动、覆盖非自己刚创建的文件，必须先取得用户确认。

## 提交与评审

- 提交信息使用 Conventional Commits，示例：`feat(frontend): 初始化水厂总览页面`。
- 一个提交只表达一个清晰目的，不混入无关格式化或重构。
- 涉及 UI 或 3D 效果的改动，应在评审说明中提供截图、录屏或明确复现步骤。
- 涉及接口契约的改动，应说明兼容性影响和前后端同步情况。

## 输出偏好

- 面向团队协作的说明优先使用中文。
- 回答要直接说明实际改了什么、怎么验证、还有哪些风险或待办。
- 不要把计划当成果；能验证的内容要实际验证。
